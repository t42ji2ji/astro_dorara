## 這個單元你會學會做什麼

把**學員要學會 X** 這種模糊描述，轉成可觀察、可驗證的行為。

🟢 基礎 ｜ 約 15 分鐘

---

## 快速說明

「學會」是個陷阱詞。

當你說**學員要學會 Prompt Engineering**，你其實不知道自己在說什麼。學到什麼程度算**會**？怎麼知道他**會**了？

這個模糊性會讓你掉進一個陷阱：既然不知道「會」是什麼，那就把「可能有用的」都塞進去吧。結果課程越來越肥，學員越來越迷失。

Action Mapping 的第一步，是把這種模糊目標轉成行為：

> 上完課，學員能夠 _______________。

填空的內容必須符合兩個條件：

1. **可觀察**：別人能看到他在做這件事
2. **可驗證**：能判斷他做得對不對

這不只是文字遊戲。當你能清楚說出學員要「做」什麼，你就有了一把尺，可以衡量課程裡的每一個內容：「這個知識點，跟學員做到目標行為有關嗎？」沒關係的，就該刪掉。

### 為什麼這很重要

如果你連**學員要做到什麼**都說不清楚，後面的課程設計會全部走歪。你會：
- 塞太多**可能有用**的知識
- 設計不出有意義的練習
- 不知道怎麼評估學習成效

還有一個更深層的問題：你可能根本不該做課程。

Cathy Moore 的核心主張是：**培訓很少是解決問題的最佳或唯一方案**。員工表現不好，原因可能是流程太複雜、工具難用、激勵機制錯誤，這些都不是培訓能解決的。

當你被迫先定義「學員要做到什麼」，你會發現有些問題的答案是：「這不是培訓的事。」這時候，你的工作是回去跟利害關係人說：「我們需要改的是流程，不是培訓員工。」

## 練習

<Exercise title="練習：定義目標行為" level={2}>
  <Fragment slot="scenario">
    你正在規劃一堂「Claude Code 入門」課程。產品經理問你：「上完這堂課，學員能做什麼？」

    你原本想回答：「學員會了解 Claude Code 的功能和使用方式。」

    但你知道這樣太模糊了。
  </Fragment>
  <Fragment slot="task">
    把**學員會了解 Claude Code 的功能和使用方式**改寫成可觀察的行為。

    用這個句型：**上完課，學員能夠 _______________。**
  </Fragment>
</Exercise>

<Template title="你的答案">
上完課，學員能夠 ________________________________
</Template>

<Answer>
好的改寫可能是：

- 用 Claude Code 重構一段現有程式碼
- 用 Claude Code 為一個函式寫單元測試
- 用 Claude Code 找出程式碼中的 bug 並修復

這些都是可觀察、可驗證的。你能看到學員操作，也能判斷結果對不對。

<Callout type="warning">
不同的目標行為會導向完全不同的課程內容。**重構程式碼**和**寫單元測試**需要的知識不一樣。這就是為什麼第一步這麼重要。
</Callout>
</Answer>

## 常見問題

<Callout type="info" title="一堂課可以有多個目標行為嗎？">
可以，但要小心。太多目標會讓課程失焦。建議：
- 初級課程：1-2 個核心行為
- 完整課程：3-5 個核心行為，每個是一個單元
</Callout>

<Callout type="info" title="如果我教的是「觀念」，不是「技能」呢？">
問自己：學員懂了這個觀念之後，會**做**什麼不一樣的事？

比如你想教**成長心態**，觀念本身無法驗證。但你可以轉成：**遇到挫折時，能說出三個從中學到的東西**。這就是可觀察的行為了。

如果想不出任何行為，可能這個觀念不值得獨立成一堂課。考慮把它融入其他技能課程的**為什麼**段落。
</Callout>
