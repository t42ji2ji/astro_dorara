# Divio 四象限寫作指南

每種文件類型有不同的目的、結構和寫法。**不要混合**。

---

## 快速判斷表

| 讀者的問題 | 類型 | 關鍵詞 |
|-----------|------|-------|
| 「我想學這個，帶我走一遍」 | Tutorial | 學習、入門、第一次 |
| 「我要完成 X，告訴我步驟」 | How-to | 怎麼、如何、解決 |
| 「X 的參數/語法是什麼？」 | Reference | 查、列表、API |
| 「為什麼要這樣設計？」 | Explanation | 為什麼、原理、背景 |

---

## Tutorial（教學導向）

### 目的
讓讀者**從零開始，跟著做，獲得成功體驗**。

### 讀者狀態
- 對這個領域陌生
- 願意投入時間學習
- 需要信心：「我做得到」

### 核心原則
1. **保證成功**：每一步都要確保讀者不會卡住
2. **有具體成果**：結束時有東西可以展示
3. **最小化解釋**：只說「做什麼」，不說「為什麼」
4. **建立信心**：第一步要簡單，讓讀者相信自己能完成

### 結構模板

```markdown
# [做出某個具體成果]

在這個教學中，你會 [具體成果描述]。

完成後你會有 [可展示的東西]。

## 開始之前

你需要：
- [先備條件 1]
- [先備條件 2]

## 第一步：[動作]

[最簡單的第一步，確保成功]

```
[程式碼或操作指令]
```

你應該看到 [預期結果]。

## 第二步：[動作]

[下一個動作]

## ...

## 恭喜！

你已經完成了 [成果]。

你學會了：
- [技能 1]
- [技能 2]

## 下一步

現在你可以試試：
- [進階教學連結]
- [相關主題]
```

### 檢查清單
- [ ] 開頭就說「你會做出什麼」
- [ ] 第一步夠簡單，不會卡住
- [ ] 每一步都有預期結果可以對照
- [ ] 沒有「為什麼」的解釋（放到 Explanation）
- [ ] 結尾有成就感和下一步

### 常見錯誤
- ❌ 開頭花一大段解釋背景 → 讀者還沒開始就累了
- ❌ 步驟太大，中間會卡住 → 拆成更小的步驟
- ❌ 假設讀者知道太多 → 每個術語第一次出現時簡單說明

## How-to（任務導向）

### 目的
讓讀者**快速解決一個具體問題**。

### 讀者狀態
- 有具體任務要完成
- 有一定基礎，不需要從頭教
- 時間有限，想快速得到答案

### 核心原則
1. **一個問題，一個 How-to**：不要塞多個任務
2. **假設有基礎**：不需要解釋基本概念
3. **直達目標**：步驟精簡，沒有閒話
4. **靈活適應**：不是死板的「按照這個做」

### 結構模板

```markdown
# 如何 [完成某個任務]

## 問題

[一句話描述要解決什麼]

## 步驟

1. [第一步]
2. [第二步]
3. [第三步]

## 結果

完成後你會 [達成的狀態]。

## 注意事項

- [可能遇到的問題]
- [替代方案]
```

### 檢查清單
- [ ] 標題就說清楚「解決什麼問題」
- [ ] 步驟可以直接執行，不需要額外查資料
- [ ] 沒有背景解釋（放到 Explanation）
- [ ] 沒有手把手的引導（那是 Tutorial）
- [ ] 一個 How-to 解決一個問題

### 常見錯誤
- ❌ 開頭解釋太多背景 → 直接給步驟
- ❌ 步驟太詳細 → 假設讀者有基礎
- ❌ 一個 How-to 塞多個任務 → 拆開

### 與 Tutorial 的差異

| Tutorial | How-to |
|----------|--------|
| 從零開始 | 假設有基礎 |
| 學習為目的 | 完成任務為目的 |
| 手把手引導 | 精簡步驟 |
| 「帶我走一遍」 | 「告訴我怎麼做」 |

## Reference（查閱導向）

### 目的
讓讀者**快速找到精確的資訊**。

### 讀者狀態
- 知道自己要找什麼
- 需要精確資訊（參數、語法、選項）
- 可能在寫程式或做任務時來查

### 核心原則
1. **完整**：涵蓋所有選項和參數
2. **精確**：資訊要正確，不能模糊
3. **好找**：有目錄、可搜尋、結構清楚
4. **沒有教學**：不是教你怎麼用，是告訴你「有什麼」

### 結構模板

```markdown
# [API / 組件 / 工具名稱]

[一句話說明這是什麼]

## 語法

```
[語法結構]
```

## 參數

| 參數 | 類型 | 預設值 | 說明 |
|-----|------|-------|------|
| ... | ... | ... | ... |

## 回傳值

[說明回傳什麼]

## 範例

```
[基本用法範例]
```

## 相關

- [相關 API / 組件]
```

### 檢查清單
- [ ] 開頭一句話說明是什麼
- [ ] 參數/選項用表格列出
- [ ] 有範例
- [ ] 沒有引導語（「首先你需要...」）
- [ ] 資訊完整，不需要查其他地方

### 常見錯誤
- ❌ 有教學口吻 → Reference 不是教你用，是讓你查
- ❌ 資訊不完整 → 所有選項都要列
- ❌ 沒有範例 → 至少一個基本範例

## Explanation（理解導向）

### 目的
讓讀者**理解背景、脈絡、設計決策**。

### 讀者狀態
- 想深入理解
- 願意花時間閱讀
- 問「為什麼」而不是「怎麼做」

### 核心原則
1. **回答「為什麼」**：不是操作步驟，是設計理由
2. **有脈絡**：說明歷史、背景、演變
3. **多角度**：比較不同方案的優缺點
4. **連結概念**：把點連成線

### 結構模板

```markdown
# [概念/決策] 的背景

## 問題

[這個設計要解決什麼問題？]

## 歷史脈絡

[這個領域的演變，之前是怎麼做的]

## 設計決策

[為什麼選擇這個方案？]

### 考慮過的替代方案

[其他可能的做法，為什麼沒選]

### 權衡取捨

[這個決策的優點和代價]

## 與其他概念的關係

[和相關概念怎麼連結]

## 總結

[核心要點]
```

### 檢查清單
- [ ] 回答了「為什麼」
- [ ] 有歷史或背景脈絡
- [ ] 討論了替代方案和權衡
- [ ] 沒有操作步驟（那是 How-to）
- [ ] 概念之間有連結

### 常見錯誤
- ❌ 變成 How-to → Explanation 不教你做，教你理解
- ❌ 只有結論沒有理由 → 要說「為什麼」
- ❌ 沒有對比 → 討論替代方案才能看出選擇的意義

## 如何在一份講義中混合使用

一份完整的課程講義可能包含四種類型，但**每個段落/章節應該只屬於一種**。

### 組織方式

```
課程講義
├── 這堂課你會做出什麼（Tutorial 預告）
├── 快速上手（Tutorial）
│   └── 跟著做，產出成果
├── 常見任務（How-to 集合）
│   ├── 如何做 X
│   ├── 如何做 Y
│   └── 如何做 Z
├── 速查表（Reference）
│   └── 參數、語法、選項
└── 深入理解（Explanation，選讀）
    └── 為什麼這樣設計
```

### 標示方式

在講義中用標籤標示類型：

- 🎯 **Tutorial**：跟著做
- 🔧 **How-to**：任務導向
- 📖 **Reference**：查閱用
- 💡 **Explanation**：深入理解

### 常見組合

| 課程類型 | 建議組合 |
|---------|---------|
| 工具入門 | Tutorial（主）+ Reference（速查）+ How-to（進階任務） |
| 方法論 | Explanation（理論）+ Tutorial（實踐） |
| 技能培養 | Tutorial（練習）+ How-to（應用場景） |
| 專業養成 | 四種都需要，分章節組織 |
