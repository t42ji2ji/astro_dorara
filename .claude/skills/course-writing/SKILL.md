# 課程講義寫作 Skill

這個 skill 引導你完成課程講義的寫作流程，從定位到完稿。

---

## 觸發條件

當使用者：
- 說「寫課程」「寫講義」「設計教材」
- 開始撰寫新的課程單元
- 問「這段內容該怎麼寫」

---

## 執行流程

### 第一步：定位內容類型（Divio 四象限）

在寫任何內容之前，先問：**這段內容是什麼類型？**

| 類型 | 讀者狀態 | 核心問題 | 寫法 |
|-----|---------|---------|------|
| **Tutorial** | 想學習，願意跟著做 | 「帶我走一遍」 | 手把手引導，確保成功體驗 |
| **How-to** | 有具體任務，想快速解決 | 「怎麼做 X？」 | 步驟清單，直達目標 |
| **Reference** | 需要查資料，知道要找什麼 | 「X 的參數是什麼？」 | 精確、完整、可查閱 |
| **Explanation** | 想理解，願意花時間 | 「為什麼是這樣？」 | 背景脈絡、設計決策 |

**判斷方式**：問自己「讀者來看這段內容時，他想要什麼？」

詳細指南見 [DIVIO_GUIDES.md](DIVIO_GUIDES.md)

---

### 第二步：選擇教學方法論

根據**課程目標的本質**選擇教學方法：

| 目標類型 | 特徵 | 適合的方法 |
|---------|-----|-----------|
| 行為改變 | 學員要**做**不一樣的事 | Action Mapping |
| 複雜技能整合 | 多種子技能要同時運作 | 4C/ID |
| 流程合規 | 步驟必須正確，不能試錯 | Gagné 九大事件 |
| 深層理解 | 要能遷移到新情境 | UbD + Merrill |

**大多數情況**：用 **Action Mapping**（聚焦行為改變）。

詳細指南見 [TEACHING_METHODS.md](TEACHING_METHODS.md)

---

### 第三步：資源搜尋（選用）

當你需要：
- 補充專家觀點增加可信度
- 找案例或數據支撐論點
- 確認最佳實踐或常見錯誤

使用 [RESOURCE_SEARCH.md](RESOURCE_SEARCH.md) 的策略進行深度挖掘。

**搜尋原則**：
1. 原創者 > 二手詮釋
2. 有數據 > 純觀點
3. 品質 > 數量（2-3 個高品質資源比 10 個淺的好）

---

### 第四步：開始寫作

根據第一步定位的類型，使用對應的結構模板：

#### Tutorial 結構
```
1. 你會做出什麼（具體成果）
2. 開始之前需要什麼
3. 第一步...（每步都確保成功）
4. 恭喜你完成了！
5. 下一步可以試試...
```

#### How-to 結構
```
1. 問題描述（一句話）
2. 解決步驟（編號清單）
3. 完成
```

#### Reference 結構
```
1. 概述（一句話說明是什麼）
2. 參數/屬性表格
3. 範例
4. 相關項目
```

#### Explanation 結構
```
1. 問題的背景
2. 設計決策的理由
3. 與其他方案的對比
4. 總結
```

詳見 [DIVIO_GUIDES.md](DIVIO_GUIDES.md) 的完整模板。

## 寫作檢查清單

### 通用檢查
- [ ] 開頭有明確的「這是什麼 / 你會得到什麼」
- [ ] 內容類型單一（不是混合體）
- [ ] 標題讓人知道「這對我有什麼用」

### 按類型檢查

**Tutorial**：
- [ ] 讀者跟著做一定會成功
- [ ] 有具體可展示的成果
- [ ] 解釋最少化，行動最大化

**How-to**：
- [ ] 步驟可以直接執行
- [ ] 沒有多餘的背景解釋
- [ ] 解決一個具體問題

**Reference**：
- [ ] 資訊完整、精確
- [ ] 好找（有目錄、可搜尋）
- [ ] 沒有教學性質的引導

**Explanation**：
- [ ] 回答了「為什麼」
- [ ] 有脈絡和背景
- [ ] 不是操作步驟

---

## 常見錯誤

| 錯誤 | 症狀 | 修正 |
|-----|------|------|
| Tutorial 混入 Explanation | 「先解釋原理再動手」 | 刪掉原理，直接動手；原理放到選讀章節 |
| How-to 寫成 Tutorial | 步驟太詳細，假設讀者什麼都不會 | 假設讀者有基礎，只寫必要步驟 |
| Reference 有教學口吻 | 「首先你需要理解...」 | 刪掉引導語，直接列資訊 |
| Explanation 變成 How-to | 「為什麼」變成「怎麼做」 | 聚焦在設計決策和脈絡 |
