# 課程講義寫作指南（SOP 草稿 v0.7）

## 寫在前面

這份指南融合了多種經過驗證的教學設計方法論：

- **Action Mapping**：讓你聚焦在行為改變，剔除無效內容
- **Gagné 九大教學事件**：高風險、流程性內容的認知鷹架
- **4C/ID 模型**：複雜技能的整合教學
- **Merrill 首要教學原理**：問題導向的認知激活
- **Divio 四象限**：讓你知道講義該有哪些類型的內容

核心原則：**沒有萬能的方法論，選對工具比努力更重要。**

---

## 第零步：選擇適合的教學設計方法

在動筆之前，先問自己三個問題：

### 問題一：學習目標的本質是什麼？

| 目標類型 | 特徵 | 適合的方法 |
|---------|-----|-----------|
| 行為改變 | 學員要**做**不一樣的事（銷售話術、客服應對） | Action Mapping |
| 複雜技能整合 | 多種子技能要同時運作（程式設計、醫療診斷） | 4C/ID |
| 流程合規 | 步驟必須正確，不能靠試錯（設備操作、安全程序） | Gagné 九大事件 |
| 深層理解 | 要能遷移到新情境（領導力、策略思考） | UbD 逆向設計 |

### 問題二：內容的風險程度？

| 風險程度 | 可以讓學員試錯嗎？ | 教學策略 |
|---------|------------------|---------|
| 低風險 | 可以，失敗是學習的一部分 | Action Mapping 的情境模擬，讓學員在安全環境中失敗 |
| 高風險 | 不行，錯誤代價太高 | Gagné 的**提供輔導**，每一步都有指導與回饋 |

### 問題三：你的課程屬於哪種類型？

| 課程類型 | 範例 | 建議組合 |
|---------|-----|---------|
| 工具入門 | Claude Code、Cursor、Excel | Action Mapping（聚焦行為）+ Gagné（操作步驟的鷹架）|
| 方法論導讀 | 超速學習、原子習慣 | Action Mapping（轉成練習）+ UbD（確保理解能遷移）|
| 技能培養 | Prompt Engineering、寫作 | Merrill（問題導向）+ Action Mapping（大量練習）|
| 專業養成 | 全端工程師、產品經理 | 4C/ID（完整任務）+ Gagné（程序性知識）|

### 決策流程圖

```
你的課程目標是什麼？
│
├─ 改變特定行為 → Action Mapping
│
├─ 培養複雜專業技能 → 4C/ID
│   （多種子技能要整合運作）
│
├─ 教會標準流程 → Gagné 九大事件
│   （步驟不能錯、風險高）
│
└─ 建立深層理解 → UbD + Merrill
    （要能遷移到新情境）
```

**大多數情況**：你的課程是**行為改變**類型，用 Action Mapping 就對了。這份 SOP 的主要內容就是基於 Action Mapping。

如果你的課程屬於其他類型，請參考附錄的**進階方法論**段落。

## 第一步：定義這堂課要學員「做到」什麼

在動筆寫任何內容之前，先回答這個問題：

> 「上完這堂課，學員能夠 _______________。」

這句話必須是**可觀察的行為**，不是模糊的理解。

| 壞的寫法 | 好的寫法 |
|---------|---------|
| 理解 Prompt 的原理 | 寫出讓 AI 正確完成任務的 prompt |
| 了解 Action Mapping | 用 Action Mapping 設計一個練習活動 |
| 認識超速學習的概念 | 為自己規劃一個 30 天的學習專案 |

如果你寫不出「學員能做到 X」，這堂課可能不該存在，或者該拆成更小的單元。

---

## 第二步：拆解「做到」需要哪些子行為

一個大行為通常由多個小行為組成。列出來。

**範例：寫出讓 AI 正確完成任務的 prompt**

需要的子行為：
1. 判斷任務適不適合交給 AI
2. 把模糊需求轉成明確指令
3. 提供足夠的上下文
4. 指定輸出格式
5. 發現 AI 誤解時，調整 prompt 修正

這些子行為就是你課程的「章節」或「單元」。

## 第三步：為每個子行為設計練習活動

這是整份講義最重要的部分。不是先寫「知識點」，是先設計「練習」。

### 好的練習長這樣

```
【情境】
你的老闆丟給你一份 30 頁的會議紀錄，說：「整理一下重點給我。」
你決定用 AI 來幫忙。

【任務】
寫一個 prompt，讓 AI 幫你整理這份會議紀錄的重點。

【你的 prompt】
_______________________

【參考答案與解說】
（學員提交後才顯示）
```

### 壞的練習長這樣

```
【問題】
以下哪一個是好的 prompt 寫法？
A. 整理這份文件
B. 請將這份會議紀錄整理成 5 個重點，每點不超過 50 字
C. 幫我看看這個
D. 以上皆非
```

**差別在哪？**

好的練習讓學員「做」他們日後會做的事。
壞的練習讓學員「認出」正確答案，但不保證他們能自己產出。

### 參考答案的寫法

參考答案不是「標準答案」。好的參考答案包含：

1. **多個可能的答案**：讓學員知道解法不只一種
2. **為什麼這樣是好的**：說明好答案的共同特徵
3. **常見錯誤會怎樣**：學員可以對照自己有沒有犯這些錯

範例：
```
【參考答案】
以下都是合理的做法：
- git checkout -b fix-login-bug
- git switch -c fix-login-bug

好答案的特徵：
1. 建立了新分支
2. 同時切換到該分支
3. 分支名稱有意義

常見問題：
- 只寫 git branch fix-login-bug → 建了分支但沒切過去，改動還是在 main 上
- 分支名稱用 test 或 temp → 之後會忘記這個分支是幹嘛的
```

### 練習活動設計檢查表

- [ ] 有具體情境，不是抽象問題
- [ ] 學員需要「產出」而不是「選擇」
- [ ] 反饋顯示「後果」而不只是「對錯」
- [ ] 和真實使用場景相似
- [ ] 參考答案有多個可能的解法
- [ ] 參考答案說明「為什麼好」和「常見錯誤」

## 第四步：決定每個練習需要的「最少資訊」

現在才開始想：學員要完成這個練習，最少需要知道什麼？

**注意：是「最少」，不是「所有相關的」。**

常見錯誤是塞太多背景知識。學員只需要知道「夠用」的資訊，其他的放到「延伸閱讀」或「為什麼」章節。

### 資訊類型分類

| 類型 | 放哪裡 | 學員什麼時候看 |
|-----|-------|--------------|
| 完成練習必須知道的 | 練習前的簡短說明 | 做之前 |
| 可以查表的細節 | 速查表 | 做的時候 |
| 背景原理 | 「為什麼」章節 | 做完之後（選讀） |
| 進階技巧 | 進階章節 | 基礎熟練之後 |

---

## 第五步：組織講義結構

一個單元的結構：

```
## 單元標題（對應一個子行為）

### 這個單元你會學會做什麼
（一句話，可觀察的行為）

### 快速說明
（完成練習需要的最少資訊，通常 3-5 個要點）

### 練習
（情境 → 任務 → 作答區 → 參考答案）

### 常見問題
（學員做練習時會遇到的卡點）

### 下一步
（引導到下一單元，製造期待感）

### 延伸：為什麼這樣做？
（背景原理，選讀）
```

## 第六步：加入導航輔助

讓學員知道自己在哪、要去哪。

### 難度標示

- 🟢 基礎：第一次接觸也能做
- 🟡 進階：需要先完成基礎單元
- 🔴 專家：有實戰經驗後再挑戰

### 時間估計

每個單元標示大約需要的時間：「約 15 分鐘」「約 30 分鐘」

### 進度追蹤

如果是線上課程，讓學員能標記「已完成」。

## 寫作風格指南

### 標題設計

標題要讓人一眼看懂「這對我有什麼用」。

**壞標題的特徵**：
- 用術語（「定義目標行為」）
- 太抽象（「課程簡介」）
- 像教科書章節（「單元 1：XXX」）

**好標題的特徵**：
- 用問句勾起好奇（「為什麼選對了還是不會？」）
- 點出痛點（「準備了 10 頁講義，學員只需要 3 頁」）
- 用數字或對比製造張力（「選擇題 vs 實作題」）

| 壞標題 | 好標題 |
|-------|-------|
| 課程簡介 | 為什麼你的課程「教了」但學員「不會」？ |
| 定義目標行為 | 「學會」到底是什麼意思？ |
| 設計練習活動 | 選擇題 vs 實作題：為什麼選對了還是不會？ |
| 精簡知識內容 | 準備了 10 頁講義，學員只需要 3 頁 |

**關鍵字策略**：
- 把學員會搜尋的詞放進標題
- 用學員的語言，不是專家的語言
- 「Prompt 怎麼寫」比「Prompt Engineering 技巧」好

---

### 避免 AI 味

1. **少用破折號「——」**
   - ❌ 這不只是技術問題——而是關乎效率的根本。
   - ✅ 這不只是技術問題，而是關乎效率的根本。
   - ✅ 這不只是技術問題。說到底，這關乎效率。

2. **減少「不是 X，而是 Y」句型**
   - ❌ 學習不是被動吸收，而是主動建構。
   - ✅ 學習需要主動建構，光是聽講不夠。

3. **少用「首先...其次...最後...」**
   - 偶爾用可以，每段都用就很機械。

4. **避免過度修飾**
   - ❌ 這個極其重要且關鍵的核心概念
   - ✅ 這很重要

### 「」vs 粗體的使用規範

| 用途 | 使用 | 範例 |
|-----|------|-----|
| 強調關鍵概念 | `**粗體**` | 學員要能**做到**什麼 |
| 引用對話/心聲 | 「」 | 學員會想：「這跟我有什麼關係？」 |
| 超出一般經驗的隱喻 | 「」 | 「學會」是個陷阱詞 |
| 專有名詞 | `**粗體**` | 這個方法叫做 **Action Mapping** |

**判斷方式**：
- 如果是某人說的話、心裡想的 → 用「」
- 如果是出格的隱喻、需要讀者停下來想一想的 → 用「」
- 其他強調 → 用 `**粗體**`

---

### 簡潔原則

- 一個段落講一件事
- 能用表格就不用段落
- 能用清單就不用表格
- 刪掉所有**其實**、**基本上**、**事實上**

## 第七步：學員視角審視

寫完初稿後，用學員的眼睛重新讀一遍。這是最容易被跳過、但最重要的步驟。

### 審視清單

**開場是否有吸引力？**
- [ ] 學員能在 10 秒內知道「這對我有什麼用」
- [ ] 有「你會產出什麼」的具體描述（不只是「你會學到 X」）
- [ ] 標題讓人想點進去

**練習難度是否適當？**
- [ ] 第一個練習夠簡單，讓學員建立信心
- [ ] 複雜練習有「填空版」作為暖身
- [ ] 每個練習都有「自我檢核」讓學員知道自己做得對不對

**單元之間是否順暢？**
- [ ] 每個「下一步」段落有鉤子，製造期待
- [ ] 單元之間有邏輯連結，不是硬切

**結尾是否有完成感？**
- [ ] 有「你完成了什麼」的總結
- [ ] 有「下一步行動」的建議
- [ ] 學員知道自己產出了什麼具體成果

### 常見問題與修正

| 問題 | 症狀 | 修正方式 |
|-----|------|---------|
| 練習跳太大 | 學員卡住、放棄 | 加「填空版」練習作為暖身 |
| 缺乏自我檢核 | 學員不確定自己做得對不對 | 每個練習後加檢核清單 |
| 單元連結弱 | 學員讀完一個單元，不想繼續 | 「下一步」要預告痛點，製造期待 |
| 沒有成果預覽 | 學員不知道學完會得到什麼 | 開頭加「你會產出 X」 |
| 結尾沒儀式感 | 學員不知道自己「畢業」了 | 加總結 + 下一步行動 |

### 練習設計的難度梯度

對於複雜的練習，使用這個梯度：

```
Level 1：填空題
給一個 80% 完成的練習，讓學員補完關鍵部分

Level 2：改寫題
給一個「壞的」範例，讓學員改成「好的」

Level 3：從頭設計
讓學員為自己的主題設計完整練習
```

### 自我檢核模板

在每個練習後加這種檢核：

```
**自我檢核**：你的 [X]...
- [ ] 條件 1
- [ ] 條件 2
- [ ] 條件 3

如果三個都打勾，你的 [X] 就夠好了。
如果有沒打勾的，回去調整那個部分。
```

## 自我檢查清單

寫完講義後，過一遍這個清單：

### 結構面
- [ ] 每個單元開頭都有明確的「學會做什麼」
- [ ] 練習活動佔講義 50% 以上的篇幅
- [ ] 知識說明只包含完成練習的最少資訊
- [ ] 有難度標示和時間估計

### 練習面
- [ ] 每個練習都有具體情境
- [ ] 學員需要產出，不只是選擇
- [ ] 反饋說明後果，不只是對錯

### 寫作面
- [ ] 沒有連續使用破折號
- [ ] 沒有過多「不是 X 而是 Y」
- [ ] 段落簡短，每段一個重點
- [ ] 刪掉了多餘的修飾詞

## 附錄：常見課程類型的調整

### 工具教學類（Claude Code、Cursor 等）

- 操作步驟會過時，和概念分開放
- 每個功能用「什麼情況下用」來組織，不是「功能列表」
- 加入「出錯了怎麼辦」的 troubleshooting

### 方法論類（超速學習、學習如何學習等）

- 原書內容是「知識」，你要轉成「練習」
- 讓學員把方法套用在自己的案例上
- 設計「規劃模板」讓學員填寫

### 技能類（Prompt Engineering、寫作等）

- 大量範例對比（好 vs 壞）
- 練習要讓學員產出，然後自己評估
- 提供評估標準，讓學員能自我檢查

## 附錄：進階教學設計方法論

當 Action Mapping 不夠用時，參考以下方法。

### Gagné 九大教學事件

適用於：高風險操作、合規培訓、技術流程。

統合分析數據（醫療教育領域，825 名參與者）：
- 知識考試成績：效應量 1.55（極大）
- 實踐技能：效應量 1.83
- 學習滿意度：是傳統教學的 8 倍

**九個步驟**：

1. **引起注意**：用震撼數據或問題開場
2. **告知目標**：明確說明學完能做什麼
3. **喚起舊知**：連結學員已知的東西
4. **呈現教材**：展示正例與反例
5. **提供輔導**：給予認知鷹架，不讓學員在黑暗中摸索
6. **引發表現**：讓學員實際操作
7. **提供回饋**：立即、具體的回饋
8. **評估表現**：確認學員達到標準
9. **促進遷移**：連結到真實工作場景

**與 Action Mapping 的差異**：Action Mapping 鼓勵讓學員先試錯再學習；Gagné 在第 5 步「提供輔導」確保學員不會建立錯誤的心理模型。高風險內容用 Gagné。

---

### 4C/ID（四元素教學設計）

適用於：培養專家級的複雜技能（如程式設計、醫療診斷、飛行訓練）。

**四個元素**：

1. **學習任務（Learning Tasks）**
   - 從一開始就給完整的真實任務，不是拆成碎片
   - 任務按難度分級，從簡單到複雜
   - 提供鷹架支持，隨著進程逐漸撤除

2. **支持性資訊（Supportive Information）**
   - 對應「非重複性技能」（推理、問題解決）
   - 在任務開始前提供
   - 幫助學員建立心理模型

3. **程序性資訊（Procedural Information）**
   - 對應「重複性技能」（標準操作步驟）
   - Just-in-time：做的時候才給，不要事先背
   - 像工具提示、操作手冊

4. **部分任務練習（Part-task Practice）**
   - 針對必須自動化的子技能進行重複練習
   - 例如：打字速度、急救按壓頻率

**與 Action Mapping 的差異**：Action Mapping 把大目標拆成幾個行為；4C/ID 從一開始就讓學員接觸完整任務，然後逐步提高難度。專家養成用 4C/ID。

---

### Merrill 首要教學原理

適用於：需要認知激活與深度理解的課程。

**五個原理**（以「問題」為核心）：

1. **問題導向**：學習圍繞真實問題展開，問題從簡單到複雜排列
2. **激活舊知**：把新知識錨定在學員已知的東西上
3. **展示（Show Me）**：示範怎麼做，包含正例與反例
4. **應用（Let Me）**：讓學員練習，並提供逐漸撤除的指導
5. **整合**：讓學員把新技能融入日常，包括反思與公開展示

**實證數據**：
- 醫學生超音波技能培訓：技能遷移顯著優於傳統組
- 高中生物課程：創造力四個維度均顯著提升

**與 Action Mapping 的差異**：Merrill 更強調「展示」階段，不只是讓學員試錯。適合需要先看示範才能模仿的技能。

---

### 方法論組合的實務建議

**最常見的組合**：

1. **Action Mapping + Gagné**
   - 用 Action Mapping 確定「教什麼」（目標行為）
   - 用 Gagné 九事件設計「怎麼教」（教學流程）
   - 適合：工具培訓、流程培訓

2. **Action Mapping + Merrill**
   - 用 Action Mapping 設計情境練習
   - 用 Merrill 確保有足夠的「展示」和「激活」
   - 適合：技能培訓、軟技能

3. **4C/ID 為主框架**
   - 大型專業養成計畫（如 Bootcamp）
   - 在每個單元內可以用 Gagné 或 Merrill 的原則
   - 適合：程式設計、設計、醫療
